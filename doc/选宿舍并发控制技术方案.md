# 选宿舍并发控制技术方案

## 一、问题背景

选宿舍场景下，多个学生可能同时抢同一个床位。原始代码存在竞态条件：

```csharp
// 原始代码（有并发问题）
public static bool AllocateBed(int bedId, int studentId)
{
    // 步骤1：检查学生是否已有床位
    string checkSql = "SELECT COUNT(*) FROM Beds WHERE StudentId=@StudentId AND Status=1";
    int count = DBHelper.ExecuteScalar(checkSql, ...);
    if (count > 0) return false;

    // 步骤2：分配床位
    string sql = "UPDATE Beds SET StudentId=@StudentId, Status=1 WHERE Id=@Id AND Status=0";
    return DBHelper.ExecuteNonQuery(sql, ...) > 0;
}
```

**竞态条件**：步骤1和步骤2之间没有原子性保证。两个请求同时通过步骤1检查后，都可能在步骤2成功分配同一个床位。

```
时间线    学生A                          学生B
──────────────────────────────────────────────────
T1       SELECT COUNT(*) → 0 ✓
T2                                      SELECT COUNT(*) → 0 ✓
T3       UPDATE → 成功
T4                                      UPDATE → 也成功?!（竞态条件）
```

虽然 `WHERE Status=0` 能防止覆盖已分配床位，但不能阻止两人同时通过检查。

## 二、方案选型

### 2.1 候选方案

| 方案 | 核心思路 | 复杂度 | 可靠性 | 性能 |
|------|----------|--------|--------|------|
| A. 应用层 lock | C# 全局锁 | 低 | 极差（单进程有效） | 差 |
| B. 单一 UPDATE 原子操作 | WHERE 条件保证原子性 | 低 | 高 | 极好 |
| C. 事务 + SELECT FOR UPDATE | 行级排他锁 | 中 | 最高 | 较差 |
| D. 乐观锁 + 版本号 | Version 字段冲突检测 | 中 | 高 | 好 |
| E. UNIQUE 约束 + 异常捕获 | 辅助表唯一索引 | 高 | 最高 | 好 |

### 2.2 方案分析

**方案 A：应用层 lock**
```csharp
private static readonly object _lock = new object();
lock (_lock) { /* 检查+更新 */ }
```
- ✗ `lock` 仅在单进程内有效，IIS 多工作进程/多服务器部署时无效
- ✗ 全局锁串行化所有请求，性能差
- **结论：Web 场景下不可用**

**方案 C：事务 + SELECT FOR UPDATE**
```sql
START TRANSACTION;
SELECT Status FROM Beds WHERE Id=@BedId FOR UPDATE;  -- 行级锁
UPDATE Beds SET ... WHERE Id=@BedId AND Status=0;
COMMIT;
```
- ✓ 排他锁保证严格串行，可靠性最高
- △ 需要新增 `ExecuteInTransaction` 泛型方法
- △ 锁等待降低吞吐量
- **结论：业务复杂时的升级方案**

**方案 D：乐观锁 + 版本号**
```sql
UPDATE Beds SET ..., Version=Version+1 WHERE Id=@BedId AND Version=@OldVersion;
```
- ✓ 无锁等待
- △ 需要给 Beds 表加 Version 列
- △ 读-写之间有窗口（被版本号挡住）
- **结论：适合高读低写场景，本场景写多**

**方案 E：UNIQUE 约束**
- ✓ 数据库层面绝对保证
- ✗ 需要新增辅助表，改动过大
- **结论：过度设计**

### 2.3 最终选择：方案 B

**理由**：
- 改动最小：仅修改 `DormBLL.cs` 一个方法
- 无需改 DBHelper、无需事务、无需改表结构
- `UPDATE WHERE Status=0` 是数据库层面的原子操作，可靠性高
- 性能最优：无锁等待，无事务开销

## 三、方案 B 详细设计

### 3.1 核心原理

将"检查 + 更新"合并为一条 SQL，利用 `UPDATE ... WHERE` 的原子性：

```sql
UPDATE Beds 
SET StudentId=@StudentId, Status=1, AllocateTime=NOW() 
WHERE Id=@BedId AND Status=0;
```

- `affected_rows = 1`：分配成功（Status 原来是 0，现在更新为 1）
- `affected_rows = 0`：分配失败（Status 已经不是 0，说明已被他人占用）

这是数据库层面的原子操作，不需要额外的锁或事务。

### 3.2 完整流程

```
┌─────────────────────────────────────────────────────────┐
│  第一步：原子分配                                        │
│  UPDATE Beds SET ... WHERE Id=@BedId AND Status=0       │
│                                                         │
│  affected=0 → 床位已被占 → 返回 false                    │
│  affected=1 → 继续第二步                                 │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  第二步：检查学生重复床位                                │
│  SELECT COUNT(*) FROM Beds WHERE StudentId=@Sid AND Status=1 │
│                                                         │
│  count=1 → 正常分配 → 返回 true                         │
│  count>1 → 极端并发，有重复 → 进入第三步                 │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  第三步：补偿回滚                                        │
│  UPDATE Beds SET StudentId=NULL, Status=0 WHERE Id=@BedId │
│                                                         │
│  撤销本次分配 → 返回 false                               │
└─────────────────────────────────────────────────────────┘
```

### 3.3 代码实现

```csharp
public static bool AllocateBed(int bedId, int studentId)
{
    // 1. 原子分配：UPDATE WHERE Status=0 保证同一床位不会被重复分配
    string sql = "UPDATE Beds SET StudentId=@StudentId, Status=1, AllocateTime=NOW() WHERE Id=@Id AND Status=0";
    MySqlParameter[] parameters = new MySqlParameter[]
    {
        new MySqlParameter("@StudentId", studentId),
        new MySqlParameter("@Id", bedId)
    };
    int affected = DBHelper.ExecuteNonQuery(sql, parameters);

    if (affected == 0) return false; // 床位已被占或不存在

    // 2. 检查学生是否有重复床位（防极端并发）
    string check = "SELECT COUNT(*) FROM Beds WHERE StudentId=@StudentId AND Status=1";
    int count = Convert.ToInt32(DBHelper.ExecuteScalar(check, new MySqlParameter[] {
        new MySqlParameter("@StudentId", studentId)
    }));

    if (count > 1)
    {
        // 有重复，回滚本次分配
        DBHelper.ExecuteNonQuery("UPDATE Beds SET StudentId=NULL, Status=0, AllocateTime=NULL WHERE Id=@Id",
            new MySqlParameter[] { new MySqlParameter("@Id", bedId) });
        return false;
    }

    return true;
}
```

### 3.4 并发场景验证

**场景1：两人同时抢同一床位**

```
时间线    学生A                                          学生B
──────────────────────────────────────────────────────────────────
T1       UPDATE Beds SET ... WHERE Id=1 AND Status=0
T2       (affected=1, 进入第二步)                        UPDATE Beds SET ... WHERE Id=1 AND Status=0
T3       SELECT COUNT(*) → 1 ✓                          (affected=0, 床位已被占 → 返回 false)
T4       返回 true ✅                                    返回 false ❌（正确）
```

**场景2：学生同时开两个浏览器抢**

```
时间线    浏览器A                                         浏览器B
──────────────────────────────────────────────────────────────────
T1       UPDATE ... WHERE Id=1 AND Status=0 → 成功
T2       SELECT COUNT(*) → 1 ✓
T3       返回 true ✅                                    UPDATE ... WHERE Id=2 AND Status=0 → 成功
T4                                                         SELECT COUNT(*) → 2 > 1!
T5                                                         回滚 Id=2 → 返回 false ❌（正确）
```

**场景3：批次已结束**

```
前端校验：批次 Status != 1 → 提示"该批次已结束"
后端校验：grab-dorm.aspx.cs 在调用 AllocateBed 前再次检查批次状态
```

## 四、前端配合

### 4.1 选宿页面（grab-dorm.aspx.cs）

```csharp
protected void btnGrab_Click(object sender, EventArgs e)
{
    // 1. 校验批次状态
    DataTable batchDt = BatchBLL.GetBatchById(batchId);
    if (Convert.ToInt32(batchDt.Rows[0]["Status"]) != 1)
    {
        ShowToast("该批次已结束", "error");
        return;
    }

    // 2. 检查学生是否已有床位
    if (DormBLL.HasBed(studentId))
    {
        ShowToast("您已有宿舍，不能重复选择", "error");
        return;
    }

    // 3. 分配床位
    bool ok = DormBLL.AllocateBed(bedId, studentId);
    if (ok)
    {
        ShowToast("选宿成功！", "success");
        // 跳转首页
    }
    else
    {
        ShowToast("该床位已被他人选择，请换一个", "error");
        LoadRooms(); // 刷新房间列表，更新床位状态
    }
}
```

### 4.2 用户体验

| 场景 | 提示信息 | 后续操作 |
|------|----------|----------|
| 分配成功 | "选宿成功！即将跳转到首页..." | 自动跳转 |
| 床位被抢 | "该床位已被他人选择，请换一个" | 刷新房间列表 |
| 已有宿舍 | "您已有宿舍，不能重复选择" | 不显示房间列表 |
| 批次结束 | "该批次已结束" | 跳回批次列表 |

## 五、边界情况处理

| 场景 | 处理方式 |
|------|----------|
| 数据库连接断开 | 事务自动回滚（InnoDB 默认行为），无脏数据 |
| 学生刷新页面重复提交 | POST-Redirect-GET 模式，刷新安全 |
| 床位状态不一致 | 第二步检查 + 补偿回滚，保证最终一致 |
| 高并发锁等待 | 方案B无锁，不存在等待问题 |

## 六、升级路径

如方案 B 不能满足未来需求（如业务逻辑变得更复杂），可升级到方案 C：

```csharp
// 方案C：事务 + FOR UPDATE
public static bool AllocateBed(int bedId, int studentId)
{
    return DBHelper.ExecuteInTransaction((conn, tx) =>
    {
        // 1. 锁定行
        MySqlCommand cmd1 = new MySqlCommand(
            "SELECT Status FROM Beds WHERE Id=@Id FOR UPDATE", conn, tx);
        // ...
        // 2. 检查 + 更新
        // ...
    });
}
```

需要新增 `DBHelper.ExecuteInTransaction` 泛型方法。
